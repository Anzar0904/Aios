## 2. Development Environment Setup

To bootstrap the local development environment:

1. **System Prerequisites**:
   * Python 3.10+ installed.
   * Git installed.
   * `uv` package manager installed (recommended for fast package builds).
2. **Virtual Environment Construction**:
   * Navigate to the project root and build the virtual environment:
     ```bash
     python3 -m venv .venv
     source .venv/bin/activate
     ```
3. **Dependency Provisioning**:
   * Install the core packages in editable development mode alongside quality checker tools:
     ```bash
     pip install -e ./core pytest ruff
     ```
4. **Validation Run**:
   * Run the test suite and verify that all tests pass successfully:
     ```bash
     PYTHONPATH=. pytest
     ```
   * Confirm that the linter runs clean:
     ```bash
     ruff check ./core
     ```

## 3. Git Branching & Commit Conventions

### 3.1 Branching Strategy
* **`main` Integrity**: The `main` branch is protected. Direct commits to `main` are prohibited.
* **Short-Lived Branches**: All updates must be implemented in short-lived feature branches branched from `main`.
* **Branch Naming Standard**: Name branches using lowercase words separated by hyphens, prefixed by feature classification:
  * `feat/add-github-tool`
  * `fix/session-leak`
  * `refactor/event-bus-handlers`
  * `docs/update-security-guide`
* **Post-Merge Housekeeping**: Branch histories are merged using squashed commits, and the remote/local feature branch must be deleted immediately after merge.

### 3.2 Commit Message Conventions
All commit messages must adhere to the Conventional Commits specification:

```
[type]([optional-scope]): [description]

[optional body]

[optional trailer]
```

* **Types**:
  * `feat`: A new feature or capability.
  * `fix`: A bug resolution.
  * `refactor`: Structural code edits with zero behavior changes.
  * `docs`: Documentation updates.
  * `test`: Adding or modifying tests.
  * `chore`: Package configurations or build scripts updates.
* **AI-Authored Commit Marking**: If the commit was written or assisted by an AI agent, the commit trailer must contain the appropriate attribution:
  `Co-authored-by: AI-agent <assistant@personal-ai-os.local>`
* *Example*:
  ```
  feat(tool): add filesystem containment validator

  Validate target paths against active workspace root dynamically.

  Co-authored-by: AI-agent <assistant@personal-ai-os.local>
  ```

## 4. Documentation Contribution Workflow

Documentation must be updated in the same commit as code changes:
* **Inline Docstrings**: All public methods and classes must include Google-style docstrings (specifying args, returns, and exceptions).
* **Metadata Blocks**: All Markdown documents inside `docs/` must begin with the standard metadata block (specifying Purpose, Scope, Related Docs).
* **Cross-Reference Links**: Hyperlinks must use relative/absolute paths utilizing the `file:///` scheme and must **never** be wrapped inside backticks.

## 5. Security & Testing Requirements

### 5.1 Quality Gates
* **Code Coverage**: Modified files must maintain at least **85%** test coverage (checked via `pytest --cov`).
* **Test Isolation**: Assert against public contracts and use `tmp_path` fixtures for filesystem transactions. Bypassing test failures is banned.

### 5.2 Security Boundaries
* **Secrets check**: Do not commit keys, credentials, or personal settings.
* **Path Validation**: Filesystem operations must validate path containment against workspace roots.
* **Command whitelists**: Subprocess commands are executed with `shell=False` and must pass whitelist validations.

## 6. AI-Assisted Contribution Guidelines

AI coding agents (including Claude) contributing code to the system must adhere to these guidelines:
1. **State Assumptions**: If a task is ambiguous, state assumptions clearly before editing code, then proceed with a logical default rather than stalling.
2. **Adhere to Code Conventions**: AI agents must respect existing code layouts, parameters constraints (max 4 parameters), function complexities, and file size limits (max 400 lines). Do not rewrite existing modules from scratch if incremental refactoring is possible.
3. **No Redundant Dependencies**: AI agents are prohibited from adding new dependencies, libraries, or external APIs without explicit user approval logged in [10_DECISION_LOG.md](file:///Users/anzarakhtar/aios/docs/10_DECISION_LOG.md).
4. **Immediate Tests Generation**: Test suites verifying new logic must be created in the same turn the logic is written.

## 7. Contributor Code of Conduct
As a project built for exactly one user, the Code of Conduct is focused on **Technical Honesty and Discipline**:
* Be direct and precise.
* Challenge assumptions constructively when planning architectures.
* Do not compromise on code safety, path validations, or test coverage.
* Respect the system boundaries: extend through the service registry, do not bypass contracts.

## 8. Common Mistakes to Avoid
* **Floating dependency tags**: Declaring versions with `^` or `~` ranges.
* **Scattered instantiations**: Importing concrete services (e.g. `LocalEventBus`) directly inside class submodules rather than utilizing composition roots.
* **Silent Error Swallowing**: Using empty `except:` blocks that hide runtime failures.
* **Markdown Link Formatting**: Wrapping relative file links inside backticks, which breaks HTML rendering.
* **Monolithic Classes**: Allowing python scripts to exceed the 400-line budget limit.
