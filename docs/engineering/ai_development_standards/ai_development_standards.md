## 1. Introduction to AI-Assisted Development

The Personal AI OS is designed to be developed *by* and *for* AI coding agents working in partnership with a human developer. Because AI agents generate much of the codebase, they must follow strict development standards to keep the system maintainable and prevent code duplication.

---

## 2. Governance Boundaries

To prevent architectural decay, AI-generated code must adhere to the following constraints:
* **No Speculative Generality**: Do not write placeholder interfaces or classes for non-existent future features. Implement only code that directly satisfies the user's active requirements.
* **Boring by Default**: Prioritize using Python's standard libraries and existing workspace modules. Do not import third-party packages unless the user explicitly requests them.
* **Reuse Existing Utilities**: Before implementing a helper function (such as for file matching or command parsing), check the workspace to see if an equivalent utility is already available.

---

## 3. Transparency & Tracing Standards

All modifications performed by AI agents must leave a clear audit trail:
* **Trace Logs**: Subsystem executions (e.g. Brain planning, tool runs) must output detailed logs to the `RuntimeIntelligenceService` and save logs under `.aios_tasks/` or `.aios_conversations/`.
* **Git Commit Tagging**: Commits generated by AI agents must follow the Conventional Commits format and include the AI co-authorship metadata:
  
  ```text
  feat: add Redis cached session store

  Co-authored-by: AI-agent <assistant@personal-ai-os.local>
  ```
  
* **Telemetry Forwarding**: All agent activities must emit lifecycle events (e.g., `AgentStartedEvent`, `AgentCompletedEvent`) to the Event Bus, allowing the system to track agent state in real time.
