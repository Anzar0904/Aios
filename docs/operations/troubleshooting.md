# Troubleshooting Guide

> Common issues, root causes, and step-by-step solutions.

## Quick Diagnostics

Run these commands first to establish baseline status:

```bash
# AIOS status
systemctl status aios

# PostgreSQL
psql -U $POSTGRES_USER -d $POSTGRES_DATABASE -c 'SELECT version();'

# Redis
redis-cli -h $REDIS_HOST -p $REDIS_PORT ping

# Qdrant
curl http://$QDRANT_HOST:$QDRANT_PORT/health

# n8n (if configured)
curl $N8N_SERVER_URL/healthz

# Recent AIOS logs
tail -100 $AIOS_HOME/logs/aios.log
```

## Common Issues

### Issue 1: AIOS fails to start with database connection error

**Root Cause**: PostgreSQL is not running or POSTGRES_* environment variables are incorrect

**Solution**:

1. Verify PostgreSQL is running: `docker-compose ps postgres`
2. Check POSTGRES_HOST, POSTGRES_DATABASE, POSTGRES_USER, POSTGRES_PASSWORD
3. Test connection manually: `psql -U $POSTGRES_USER -d $POSTGRES_DATABASE`
4. Check PostgreSQL logs: `docker-compose logs postgres`

**Related logs**:

- `aios.log`
- `postgres.log`

**See also**:

- [configuration.md#postgresql-configuration](configuration.md#postgresql-configuration)

---

### Issue 2: Redis connection refused or timeout on startup

**Root Cause**: Redis is not running, wrong host/port, or authentication failure

**Solution**:

1. Verify Redis is running: `docker-compose ps redis`
2. Check REDIS_HOST and REDIS_PORT environment variables
3. Test connection: `redis-cli -h $REDIS_HOST -p $REDIS_PORT ping`
4. If using auth, verify REDIS_PASSWORD is correct
5. Check TLS settings: REDIS_TLS should match server config

**Related logs**:

- `aios.log`
- `redis.log`

**See also**:

- [configuration.md#redis-configuration](configuration.md#redis-configuration)

---

### Issue 3: Qdrant connection error or vector collection missing

**Root Cause**: Qdrant is not running, wrong host/port, or collection not initialized

**Solution**:

1. Verify Qdrant is running: `docker-compose ps qdrant`
2. Check QDRANT_HOST and QDRANT_PORT environment variables
3. Test health: `curl http://$QDRANT_HOST:$QDRANT_PORT/health`
4. List collections: `curl http://$QDRANT_HOST:$QDRANT_PORT/collections`
5. If QDRANT_API_KEY is set, verify it matches the server config

**Related logs**:

- `aios.log`
- `qdrant.log`

**See also**:

- [configuration.md#qdrant-configuration](configuration.md#qdrant-configuration)

---

### Issue 4: Slow response times or AI model timeouts

**Root Cause**: OmniRoute provider latency, Redis cache miss, or Qdrant vector search overload

**Solution**:

1. Check OmniRoute provider health: review aios.log for provider errors
2. Verify API keys: OPENROUTER_API_KEY, ANTHROPIC_API_KEY, GEMINI_API_KEY
3. Check Redis cache hit rate: `redis-cli info stats | grep keyspace`
4. Check Qdrant search latency via /collections endpoint
5. Review provider-specific rate limits and quotas

**Related logs**:

- `aios.log`
- `redis.log`
- `qdrant.log`

**See also**:

- [configuration.md#omniroute--ai-provider-configuration](configuration.md#omniroute--ai-provider-configuration)

---

### Issue 5: Memory leaks or high memory usage

**Root Cause**: Unclosed database connections or large embedding cache

**Solution**:

1. Monitor PostgreSQL connection pool size
2. Monitor Redis memory: `redis-cli info memory`
3. Adjust REDIS_TIMEOUT to reclaim idle connections
4. Check Qdrant vector collection sizes
5. Restart services in order: AIOS Core → Redis → Qdrant → PostgreSQL

**Related logs**:

- `aios.log`
- `system.log`

**See also**:

- [monitoring.md#database-metrics](monitoring.md#database-metrics)

---

### Issue 6: n8n workflows not triggering or integration errors

**Root Cause**: n8n server not running, N8N_* variables incorrect, or authentication failure

**Solution**:

1. Verify n8n is running: `docker-compose ps n8n` or `curl $N8N_SERVER_URL/healthz`
2. Check N8N_SERVER_URL points to the correct server
3. Verify authentication: N8N_API_KEY (preferred) or N8N_EMAIL + N8N_PASSWORD
4. Check n8n logs: `docker-compose logs n8n`
5. In AIOS, run the `/n8n-status` command to diagnose connection

**Related logs**:

- `aios.log`
- `n8n.log`

**See also**:

- [configuration.md#n8n-optional-configuration](configuration.md#n8n-optional-configuration)

---

### Issue 7: Database migrations fail on startup

**Root Cause**: Schema version mismatch, insufficient permissions, or PostgreSQL unavailable

**Solution**:

1. Ensure PostgreSQL is running before running migrations
2. Verify POSTGRES_USER has CREATE TABLE and ALTER TABLE permissions
3. Run migrations manually: `python -m aios.migrations`
4. Check migration logs for specific SQL errors
5. If needed, reset schema: drop and recreate the database, then re-run migrations

**Related logs**:

- `aios.log`
- `postgres.log`

**See also**:

- [startup.md#step-4-database-migrations](startup.md#step-4-database-migrations)

## General Troubleshooting Steps

1. **Check logs**: Review `$AIOS_HOME/logs/aios.log` and service-specific logs
2. **Verify configuration**: Confirm all required environment variables are set
3. **Test connectivity**: Verify each service responds to its health check
4. **Check resources**: Monitor CPU, memory, and disk with `top`, `free`, `df -h`
5. **Review recent changes**: `git log --oneline -20` for recent deployments
6. **Restart in order**: Stop AIOS Core, then restart services per [startup.md](startup.md)
