# Backup & Restore Guide

> Procedures for backing up and restoring AIOS data.

## Backup Strategy

AIOS requires backing up the following components:

| Component | Type | Frequency | Retention | Tool |
|-----------|------|-----------|-----------|------|
| PostgreSQL Database | database | daily | 30 days | pg_dump |
| Qdrant Vector Store | database | daily | 30 days | Qdrant Snapshot API |
| Configuration Files | configuration | weekly | 90 days | tar |
| User Data & Workspaces | files | daily | 30 days | rsync |

### PostgreSQL Database

- **Type**: database
- **Backup location**: `/var/backups/aios/postgres`
- **Frequency**: daily
- **Retention**: 30 days
- **Tool**: pg_dump
- **Notes**: Contains all workspaces, sessions, engineering tasks, planning, approval, review, documentation metadata, and AI memory persistence.

### Qdrant Vector Store

- **Type**: database
- **Backup location**: `/var/backups/aios/qdrant`
- **Frequency**: daily
- **Retention**: 30 days
- **Tool**: Qdrant Snapshot API
- **Notes**: Contains conversation_memory, semantic search indexes, and runtime intelligence vectors. Use the Qdrant snapshot API.

### Configuration Files

- **Type**: configuration
- **Backup location**: `/var/backups/aios/config`
- **Frequency**: weekly
- **Retention**: 90 days
- **Tool**: tar
- **Notes**: Includes config/config.toml, .env, docker-compose files. Never store secrets in the backup archive unencrypted.

### User Data & Workspaces

- **Type**: files
- **Backup location**: `/var/backups/aios/data`
- **Frequency**: daily
- **Retention**: 30 days
- **Tool**: rsync
- **Notes**: Includes user conversation histories, workspace snapshots, and any generated artifacts stored on the filesystem.

## Backup Procedures

### PostgreSQL Backup

**Manual backup:**

```bash
pg_dump -U $POSTGRES_USER -d $POSTGRES_DATABASE -F c \
  -f /var/backups/aios/postgres/aios_$(date +%Y%m%d_%H%M%S).dump
```

**Automated backup (cron — runs at 02:00 daily):**

```bash
0 2 * * * /opt/aios/scripts/backup-postgres.sh
```

**Verify backup integrity:**

```bash
pg_restore --list /var/backups/aios/postgres/aios_20260706_020000.dump | head -20
```

### Qdrant Backup

> Qdrant collections used by AIOS include `conversation_memory` and other runtime intelligence collections.

**Create snapshot:**

```bash
curl -X POST http://$QDRANT_HOST:$QDRANT_PORT/collections/conversation_memory/snapshots
```

**Download snapshot:**

```bash
curl http://$QDRANT_HOST:$QDRANT_PORT/collections/conversation_memory/snapshots/{snapshot_name} \
  -o /var/backups/aios/qdrant/conversation_memory_$(date +%Y%m%d).snapshot
```

**List existing snapshots:**

```bash
curl http://$QDRANT_HOST:$QDRANT_PORT/collections/conversation_memory/snapshots
```

### Configuration Backup

**Backup configuration files (never include secret values in the archive):**

```bash
tar -czf /var/backups/aios/config/config_$(date +%Y%m%d).tar.gz \
  config/config.toml docker-compose.yml docker-compose.prod.yml
# Secrets (.env) are stored separately in a secrets manager
```

## Restore Procedures

### PostgreSQL Restore

```bash
# Stop AIOS Core first
systemctl stop aios

# Drop and recreate the database
dropdb -U $POSTGRES_USER $POSTGRES_DATABASE
createdb -U $POSTGRES_USER $POSTGRES_DATABASE

# Restore from backup
pg_restore -U $POSTGRES_USER -d $POSTGRES_DATABASE \
  /var/backups/aios/postgres/aios_20260706_020000.dump
```

### Qdrant Restore

```bash
# Upload snapshot to Qdrant
curl -X POST http://$QDRANT_HOST:$QDRANT_PORT/collections/conversation_memory/snapshots/upload \
  --data-binary @/var/backups/aios/qdrant/conversation_memory_20260706.snapshot

# Recover the collection from the uploaded snapshot
curl -X PUT http://$QDRANT_HOST:$QDRANT_PORT/collections/conversation_memory/snapshots/recover \
  -H 'Content-Type: application/json' \
  -d '{"location": "http://$QDRANT_HOST:$QDRANT_PORT/collections/conversation_memory/snapshots/{snapshot_name}"}'
```

### Configuration Restore

```bash
tar -xzf /var/backups/aios/config/config_20260706.tar.gz -C /opt/aios/
# Restore secrets from your secrets manager separately
```

## Disaster Recovery

In case of complete system failure:

1. Provision new infrastructure (see [deployment.md](deployment.md))
2. Install AIOS from source or deployment package
3. Restore configuration files and secrets
4. Restore PostgreSQL database from latest backup
5. Restore Qdrant vector collections from snapshots
6. Run database migrations to verify schema integrity
7. Start services in correct order (see [startup.md](startup.md))
8. Verify system health and resume monitoring

**Estimated Recovery Time Objective (RTO)**: 2–4 hours
**Recovery Point Objective (RPO)**: 24 hours (daily backups)
