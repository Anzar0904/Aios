# Service Startup Guide

> Correct service startup order and dependency management.

## Bootstrap Architecture

The AIOS startup sequence is orchestrated by `bootstrap_kernel()` in
`core/src/aios/bootstrap_modules/bootstrap_kernel.py`. The kernel constructs services in this order:

```
1. ServiceRegistry + Config
2. EventBus
3. SQL Persistence Platform (→ requires PostgreSQL)
4. Redis Platform (→ requires Redis)
5. Qdrant/Vector Memory Platform (→ requires Qdrant)
6. Runtime Services & Agents
7. Source Control Platform
8. n8n Platform (optional)
9. Kernel instantiation
```

## Startup Sequence

Start services in the following order to respect bootstrap dependencies:

### Step 1: PostgreSQL

Start PostgreSQL database server (required by Persistence Platform)

```bash
docker-compose up -d postgres
```

**Health check:**

```bash
psql -U $POSTGRES_USER -d $POSTGRES_DATABASE -c 'SELECT 1;'
```

> The AIOS Persistence Platform connects to PostgreSQL immediately on startup. Ensure POSTGRES_HOST, POSTGRES_USER, POSTGRES_PASSWORD, and POSTGRES_DATABASE are set before starting AIOS Core.

### Step 2: Redis

Start Redis cache/coordination server (required by Redis Platform)

```bash
docker-compose up -d redis
```

**Health check:**

```bash
redis-cli -h $REDIS_HOST -p $REDIS_PORT ping
```

> Redis is used for caching, session management, distributed locking, rate limiting, and inter-service queue coordination. Set REDIS_HOST and REDIS_PORT before starting AIOS Core.

### Step 3: Qdrant

Start Qdrant vector database (required by Runtime Intelligence Platform)

```bash
docker-compose up -d qdrant
```

**Health check:**

```bash
curl -sf http://$QDRANT_HOST:$QDRANT_PORT/health
```

> Qdrant stores conversation memory, semantic search indexes, and AI memory embeddings. The Runtime Intelligence Platform connects to Qdrant during bootstrap. QDRANT_HOST and QDRANT_PORT must be set.

### Step 4: Database Migrations

Run PostgreSQL schema migrations via PersistenceBootstrapper (called automatically on bootstrap_kernel initialization)

```bash
python -m aios.migrations
```

**Depends on**: PostgreSQL

> Migrations are run automatically inside bootstrap_kernel() via PersistenceBootstrapper.start(). Running manually ensures the schema is current before the first startup.

### Step 5: AIOS Core

Start the AIOS application (Kernel + all platform services). Bootstraps: ServiceRegistry → Persistence → Redis → Qdrant → RuntimeServices → SourceControl → n8n.

```bash
aios
```

**Depends on**: PostgreSQL, Redis, Qdrant

**Health check:**

```bash
aios --version
```

> The full bootstrap sequence is orchestrated by bootstrap_kernel() in bootstrap_modules/bootstrap_kernel.py. All three external services must be healthy before launching AIOS Core.

### Step 6: n8n (optional)

Start the self-hosted n8n workflow automation server. AIOS auto-detects n8n via N8N_SERVER_URL on startup.

```bash
docker-compose up -d n8n
```

**Health check:**

```bash
curl -sf $N8N_SERVER_URL/healthz
```

> n8n is optional. If N8N_SERVER_URL is not set, the n8n platform is registered but remains in a disconnected state. Authentication: set N8N_API_KEY (preferred) or N8N_EMAIL + N8N_PASSWORD.

## Automated Startup

For production, use a process manager or orchestration tool:

```bash
# Start all services with Docker Compose
docker-compose up -d

# Or using systemd (AIOS service unit)
systemctl start aios
```

## Shutdown Sequence

Shutdown in reverse startup order to prevent data loss:

```bash
# 1. Stop AIOS Core gracefully
systemctl stop aios

# 2. Stop n8n (if running)
docker-compose stop n8n

# 3. Stop external services
docker-compose stop qdrant redis postgres
```

## Startup Failures

Common startup issues:

- **PostgreSQL not ready**: Wait for PostgreSQL to accept connections before starting AIOS
- **Redis connection refused**: Verify REDIS_HOST and REDIS_PORT; check if Redis requires authentication
- **Qdrant health check fails**: Wait for Qdrant to initialize; check QDRANT_HOST
- **Migration failure**: Run `python -m aios.migrations` manually to inspect errors
- **Port already in use**: Check for conflicting services with `lsof -i :<port>`
- **API key not configured**: Ensure at least one OmniRoute provider key is set
