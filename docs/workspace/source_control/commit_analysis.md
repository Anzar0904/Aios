## 1. Commit Graph Parsing (DAG)

The repository history is modeled as a **Directed Acyclic Graph (DAG)** of commit nodes ($G_{commit} = (V_{commits}, E_{edges})$):
* **Nodes ($V_{commits}$)**: Store commit hashes, author metadata, timestamp offsets, and commit messages.
* **Edges ($E_{edges}$)**: Represent parent relationships (e.g., Commit B points to parent Commit A). Merge commits contain two parent edges.

```
   [Commit A] <--- (Parent) --- [Commit B] <--- (Parent) --- [Commit D] (Main Branch)
                                      ^
                                      | (Parent)
                                [Commit C] (Feature Branch)
```

The DAG is constructed by parsing `git log --pretty=format:"%H|%P|%an|%ae|%at|%s"` lines, loading the topology into the local SQLite graph db.

## 2. Conventional Commit Standards Validation

To maintain a clean changelog, the commit engine enforces **Conventional Commit** conventions (e.g. `feat(notion): add comments sync`).
* **Format Regex**:
  `^(feat|fix|docs|style|refactor|perf|test|build|ci|chore|revert)(?:\(([^)]+)\))?: (.{1,72})`
* **Commit Co-authorship Validation**: Ensure commits generated by the AI OS append co-authorship tags pointing to active developer profiles, maintaining attribution history.

## 3. Commit-to-File & Symbol Mapping

For each commit node, the system maps changes to specific code structures:
1. **File Modifications**: Parse `git diff-tree --no-commit-id --name-status -r [hash]` to register affected files.
2. **AST Symbol Mapping**: By parsing file diff hunks, the engine identifies the exact lines changed. It cross-references these lines with the `codebase_symbols` index to determine which classes or methods were modified (e.g., Commit `c82614` modified `class NotionProvider -> def sync_document`). This links Git change history directly to code blocks.
