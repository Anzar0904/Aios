## 1. Notion API Permission Scopes

During the OAuth 2.0 flow or configuration of internal integration tokens, the Personal AI OS requests a restricted set of capabilities. In accordance with the **Principle of Least Privilege**, the OS requests only the scopes necessary to execute current tasks.

```
       [Local Permission Checker] ===> Checks Allowed Scopes (Read/Write/Comment)
              |
              +---> Matches Scope: "read"    ===> Read Pages & Block AST
              |
              +---> Matches Scope: "write"   ===> Create/Modify Blocks & Database Properties
              |
              +---> Matches Scope: "comment" ===> Read/Write Discussion Threads
```

### Scope Matrix

| Scope ID | Notion API Capability | OS Subsystem Consumer | Purpose |
|----------|-----------------------|-----------------------|---------|
| `pages:read` | Read page metadata and block children | `MemoryService`, `KnowledgeHubService` | Syncing page content for semantic indexing |
| `pages:write` | Create pages, update block content | `ActionEngine`, `TaskExecutor` | Creating workspaces, generating code files, publishing reports |
| `databases:read` | Query database schemas and rows | `KnowledgeHub` | Syncing application tables, release trackers, logs |
| `databases:write` | Create/update rows and properties | `AutomationService`, `ApprovalEngine` | Pushing test results, updating state boards |
| `comments:read` | Retrieve comments for pages | `ReviewCollaborationService` | Syncing comment threads for review workflows |
| `comments:write` | Post comments on pages | `ReviewCollaborationService` | Appending review notes and consensus logs |

## 2. Enforcing Scopes Locally

To prevent compromised subagents or bug-prone workflows from exceeding authorizations, the AI OS enforces a **Local Scope Validation Gate**:

1. Every database or page request generated by local services is intercepted by the `NotionProvider` API handler.
2. The handler checks the active session's scope list before making a network call.
3. If an operation exceeds the authorized scope (e.g. attempting to create a database row when the session only has `pages:read` scope), the request is blocked locally, and an `UnauthorizedScopeException` is thrown:
   ```python
   class UnauthorizedScopeException(Exception):
       def __init__(self, required_scope: str, active_scopes: List[str]):
           super().__init__(
               f"Operation rejected: Requires scope '{required_scope}' "
               f"but active scopes are {active_scopes}."
           )
   ```

## 3. User Consent & Escalation

### 3.1 Workspace Containment
Notion restricts token access to folders and sub-pages manually shared with the connection. The OS cannot browse or traverse other workspaces. If a page query returns `object_not_found`, the OS displays a prompt asking the user to verify if the connection has been shared on the Notion page.

### 3.2 Escalation Scenarios
If a tool or service requests an action that exceeds the currently granted scopes:
1. The execution halts.
2. The REPL console prompts the user:
   ```
   [Notion Security Alert]
   Service 'ApprovalEngine' requests 'databases:write' permission to update the release database.
   Current scopes: ['pages:read', 'pages:write']
   
   To authorize this escalation, please re-authenticate by running:
   > notion workspace authorize --scopes pages:read pages:write databases:write
   ```
3. The OS waits for user approval and does not perform the action without explicit consent.
